# 第三方扩展不可作为编排协议

## 困境

在 R&K Flow 中希望复用 Claude Code 官方 VS Code 扩展作为 Role Chat 入口，从而减少自研聊天 UI 的维护成本。

探索后发现：官方扩展虽然可以打开、聚焦、预填 prompt，也会在本地保留部分 transcript/session 文件，但这些能力都不是公开稳定的第三方 Chat API。R&K Flow 无法稳定嵌入官方 Webview、读取内部 DOM、拦截官方聊天记录、提交 prompt 或可靠恢复会话。

如果把这些私有实现当成 TeamBus 或 Role Chat 的主协议，后续会在版本升级、隐私边界、审计完整性和多角色会话隔离上产生不可控风险。

## 策略

1. 区分“可观察到的本地实现”和“可依赖的公开 API”。
2. 对第三方扩展只使用公开命令、公开配置或显式文档化的入口。
3. 编排、审计、角色间通信必须保留在自有协议中，例如 TeamBus。
4. 用户与 Role 的私聊如果需要隔离和审计，应由自研 Role Chat 承载。
5. 第三方本地 transcript 只能作为用户显式授权后的只读导入或调试辅助，不能作为实时主数据源。
6. 后端接入应走 Adapter 抽象，MVP 先保留一个稳定后端，新增 CLI 时再扩展具体 Adapter。

## 理由

多角色 Agent 产品的核心不是“能打开一个聊天窗口”，而是“能控制会话边界、消息路由、审计记录和失败降级”。第三方官方扩展的私有 UI 和本地状态无法提供这些产品级保证。

采用自研 Role Chat + 自有 TeamBus 会增加 UI 维护成本，但可以保证 R&K Flow 的可控性、可审计性和后续扩展能力。
